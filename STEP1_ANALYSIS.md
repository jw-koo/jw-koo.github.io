# Step 1 — 개발 업무 분석

## 범위와 확인 결과

- 목표: 반응형 개인 프로페셔널 웹사이트와 상단 `Games` 메뉴의 지렁이 게임 구축
- 구현 방식: 정적 HTML, CSS, JavaScript만 사용
- 저장소 초기 상태: `README.md`만 존재하며, 애플리케이션 코드·테스트 설정은 없음
- 프로필 자료: 제공된 LinkedIn URL은 외부 확인 시 본문을 불러오지 못함. 이름, 직무, 경력, 기술, 학력, 소개 문구 등은 `[사람 확인 필요]`로 남김
- 보안: 프로필의 민감하거나 불필요한 개인정보는 수집·출력·저장하지 않음
- 이번 단계에서 하지 않는 일: 코드 수정, 테스트 실행, Git push, 배포

## 추천하는 첫 루프

가장 안전한 첫 루프는 **루프 1: 요구사항·콘텐츠 확정**이다. 아직 프로필 본문을 확인하지 못했고 코드가 없는 초기 상태이므로, 먼저 공개 가능한 자기소개와 게임 규칙을 확정하면 개인정보 노출·잘못된 경력 표기·재작업 위험을 가장 작게 유지할 수 있다.

첫 루프에서 사람에게 확인할 항목:

- 표시할 이름과 한 줄 소개: `[사람 확인 필요]`
- 직무/관심 분야/기술 스택: `[사람 확인 필요]`
- 경력·프로젝트·성과·연락처·외부 프로필 공개 범위: `[사람 확인 필요]`
- 게임의 적이 플레이어 또는 지렁이와 충돌할 때의 결과: `[사람 확인 필요]`
- 적의 등장 빈도와 난이도 상승 여부: `[사람 확인 필요]`
- 모바일 조작 방식(swipe, 화면 방향 버튼 또는 둘 다): `[사람 확인 필요]`

## 안전성 메모

- LinkedIn 본문을 읽지 못한 상태에서 경력이나 기술을 추정해 작성하지 않는다.
- 이메일·전화번호·주소·내부 계정 정보 등 민감 정보는 저장하지 않는다. 공개가 필요한 연락처도 사람 확인 후에만 넣는다.
- 랜덤 적은 `Math.random()` 자체보다 공정성, 충돌 판정, 난이도 상한, 재현 가능한 테스트 전략을 먼저 설계한다.
- 이번 산출물은 분석 문서와 `MEMORY.md`뿐이며, 구현·테스트·push·배포는 후속 승인 이후에 진행한다.
